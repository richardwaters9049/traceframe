# Technical debt resolution — 20/07/2026

## Summary

The P1–P3 debt recorded on 19/07/2026 was resolved before source ingestion.
The work established a truthful audit model, bounded data loading, stronger
authentication boundaries, repeatable migrations, accessible interaction
patterns, and broader automated verification. All fixtures and browser data
remain synthetic.

## P1 resolutions

### Audit integrity

**Changes needed:** Establish unambiguous ledger order, update the chain head
atomically, verify stored canonical events on the server, report typed integrity
states truthfully, and test corruption and concurrency scenarios.

**How it was resolved:**

- The ledger is explicitly global and uses a singleton `audit_chain_heads` row.
- Case creation locks the head with `SELECT ... FOR UPDATE`, assigns a monotonic
  `ledger_sequence`, appends the event, and advances the head in one transaction.
- A server-only verifier recalculates canonical SHA-256 digests and validates
  sequence continuity and predecessor links.
- The dashboard and case workspace render verified, broken, or unavailable
  states. A case audit panel is identified as a filtered view of the verified
  global ledger.
- Tests cover a valid chain, changed metadata, missing predecessors, reordered
  events, reversed wall-clock times, and six concurrent writes.

### Bounded dashboard and case queries

**Changes needed:** Remove eager per-case workspace queries, paginate case
summaries with a stable cursor, fetch only a selected case on demand, and test
response size under realistic volume.

**How it was resolved:**

- The dashboard loads a bounded 20-row summary page and one global audit check.
- `GET /api/cases` uses an opaque `(created_at, id)` cursor, caps page size at
  50, and returns summary fields, aggregate counts, and the next cursor.
- Authenticated `GET /api/cases/:id` loads only the selected workspace and its
  filtered audit view.
- Browser/API tests exercise pagination and enforce a response below 10 KB for
  a two-row page.

### Authentication and identity

**Changes needed:** Add privacy-conscious login throttling, case-insensitive
email uniqueness, explicit same-origin checks, fail-closed role capabilities,
and integration coverage for invalid session and abuse cases.

**How it was resolved:**

- Database-backed throttling hashes normalised account and network identity,
  applies bounded progressive delays, and temporarily blocks repeated failures.
- A migration normalises existing email addresses and enforces uniqueness on
  `lower(email)`.
- State-changing handlers require a matching `Origin`; responses and structured
  server logs carry request IDs.
- Analysts and administrators can read and create cases, reviewers are
  read-only, and unknown roles receive no case capability.
- Tests cover validation, role enforcement, origin rejection, throttling,
  revocation, generic failures, concurrent valid logins, and malformed,
  expired, revoked, or inactive-user sessions.

## P2 resolutions

### Schema, indexing, migrations, and maintenance

**Changes needed:** Add constraints and indexes for real query shapes, provide
an existing-volume migration path, schedule expired-session cleanup, and verify
the relevant query plans.

**How it was resolved:**

- Migration `0002_harden_audit_and_queries.sql` adds audit sequencing and the
  chain head, lower-email uniqueness, case checks, throttle storage, and indexes
  for case cursors, audit lookup, sessions, and future ingestion claiming.
- `EXPLAIN (ANALYZE, BUFFERS)` was run for case-page and audit-object queries.
  PostgreSQL appropriately chose sequential scans for the small local fixture;
  the supporting indexes are available as cardinality grows.
- A one-shot Compose `migrate` service records applied filenames, baselines
  legacy volumes, runs before dependent services, and fails startup on error.
  Fresh, upgraded, and repeated runs were exercised without deleting data.
- The worker performs hourly expired-session cleanup using database time.
- The README and architecture guide document immutable forward migrations,
  pre-upgrade backups, and restore or corrective-forward rollback policy.

### Accessibility and responsive layout

**Changes needed:** Implement complete modal focus behaviour, improve mobile
navigation semantics, disable unfinished controls, remove global CSS scaling,
and validate keyboard interaction, accessibility, and browser reflow.

**How it was resolved:**

- Base UI dialogs now provide inert backgrounds, focus trapping, initial focus,
  Escape dismissal, scroll locking, and opener focus restoration for the
  new-case drawer and mobile navigation.
- Sources and Relationships are disabled with explanations until their panels
  exist.
- Axe and Playwright cover login, navigation, drawer focus, case selection, and
  logout. Text contrast issues found by those checks were corrected.
- Global `zoom` and transform scaling were removed in favour of explicit
  spacing and sidebar sizing.
- Reflow checks cover viewport equivalents of 100%, 125%, 200%, and 400% zoom;
  workspace flows run in Chromium, Firefox, WebKit, and mobile Chromium.

### UI structure and client requests

**Changes needed:** Separate component responsibilities, reuse core design
tokens, safely handle unsuccessful or non-JSON HTTP responses, support aborts
and timeouts, and avoid reporting logout success before revocation.

**How it was resolved:**

- Login, form, provider, register, dialog, and content-loading concerns have
  focused components protected by browser tests.
- Repeated core surface, text, border, and state colours use named Tailwind
  theme tokens.
- A typed same-origin client helper validates status and content type, handles
  JSON and non-JSON errors, supports timeout/abort behaviour, and exposes
  request IDs.
- Logout displays success only after server-side revocation and otherwise
  presents a recoverable error.
- The shared production PostgreSQL pool is cached per process, fixing a
  connection leak exposed during cross-engine testing.

### Tests and continuous integration

**Changes needed:** Cover the highest-risk database, API, browser, accessibility,
and worker behaviour and run those checks automatically.

**How it was resolved:**

- Web unit coverage grew from 6 to 13 assertions across contracts, hashing,
  verification, and authorisation.
- API/browser tests cover authentication, validation, origin checks, concurrent
  audit writes, pagination, on-demand loading, focus, responsive navigation,
  logout, revocation, accessibility, and reflow.
- Worker settings and session maintenance have focused pytest coverage.
- GitHub Actions runs web lint/unit/build, Ruff/pytest/compile, dependency
  audits, Compose startup, and cross-engine Playwright tests.
