# Technical debt review — 19/07/2026

## Review summary

The first Traceframe vertical slice was stable: the web application linted and
built successfully, its six unit tests passed, and the Python worker compiled.
The review found no release-blocking defect for the local, single-user,
synthetic demo. It did identify structural debt that needed resolving before
source ingestion introduced more events, jobs, users, and data volume.

Priority meanings:

- **P1:** Resolve before the next major product vertical slice.
- **P2:** Resolve during the next implementation cycles.
- **P3:** Safe cleanup that improves maintainability and developer experience.

## P1 findings

### Audit integrity was asserted but not verified

The dashboard reported **Chain verified** even though the application created
hashes without recalculating stored event digests or walking every predecessor
link. The predecessor was selected by application-generated `created_at` and
`id`, which meant clock skew between future web instances could undermine the
ordering assumption. The advisory transaction lock serialised writers but did
not make timestamps monotonic.

The ledger was global, while an individual case loaded only its own events. A
case event could therefore reference an event from another case, preventing the
case screen from independently proving the complete chain.

### Dashboard loading used an N+1 query pattern

The dashboard loaded the case list and then fetched a full workspace for every
case. Rendering `N` cases produced `1 + (2 × N)` queries and sent audit data to
the client even when no case had been selected. The case register was also
unbounded and had no pagination.

### Authentication needed abuse and identity hardening

Login had no throttling or progressive delay. PostgreSQL email uniqueness was
case-sensitive even though authentication compared lower-cased addresses,
allowing ambiguous accounts through direct database writes. State-changing
handlers relied on SameSite cookies and JSON requests without explicit origin
validation.

All authenticated users could see every case, and stored roles were not
enforced. That was acceptable only for the explicitly single-user demo.

## P2 findings

### Database indexes and constraints were incomplete

Case ordering, audit-object lookup, session expiry, and future ingestion-job
claiming lacked supporting indexes. Case priority and status were validated in
TypeScript but not constrained by PostgreSQL. Expired sessions were cleaned up
only when a new session was created.

### Migrations worked reliably only for fresh volumes

Schema SQL was mounted into PostgreSQL's `docker-entrypoint-initdb.d`, which
runs only when a database volume is first initialised. Existing developer
volumes would not receive later schema changes automatically.

### Drawer and navigation accessibility was incomplete

The new-case drawer supported Escape and exposed modal semantics but did not
fully trap focus, make background content inert, or reliably restore focus to
the opener. The mobile sidebar had similar shortcomings. Sources and
Relationships appeared enabled despite having no actions.

### Layout density depended on global CSS scaling

The application used `zoom: 0.78` and a transform fallback to produce its
compact desktop composition. Those techniques can behave inconsistently for
fixed and sticky elements, focus outlines, pointer coordinates, text rendering,
and browser accessibility zoom.

### UI responsibilities and design values were repeated

Large components combined animation, networking, timers, responsive layout,
modal behaviour, and navigation. Brand colours and motion values were repeated
as arbitrary Tailwind values rather than expressed through shared tokens.

### Client request handling assumed ideal responses

Logout did not check `response.ok`, so a failed revocation could still display
success. Login and case creation assumed JSON responses, requests had no common
timeout or cancellation behaviour, and logs lacked request correlation.

### Test coverage and automation were narrow

Six unit tests covered validation contracts and deterministic hashing. There
were no database concurrency, Route Handler, browser, accessibility, or worker
tests, and no continuous-integration workflow.

## P3 findings

### Documentation and generated scaffold cleanup

The root README was current at review time, but the web-app README still
contained generic create-next-app instructions for unsupported package
managers. Architecture documentation described worker behaviour in the future
tense, and unused starter SVG assets remained in the web app.

### Local tooling was not fully reproducible

`make test` assumed an existing Python virtual environment. There was no common
bootstrap command for dependencies and tool checks, and Python verification
stopped at bytecode compilation.

## Recommended resolution order

1. Make audit verification truthful and structurally reliable.
2. Replace eager workspace loading with pagination and on-demand retrieval.
3. Harden login, identity, request-origin checks, and authorisation.
4. Establish repeatable migrations, indexes, constraints, and maintenance.
5. Add database/API integration tests and continuous integration.
6. Complete dialog accessibility and remove misleading controls.
7. Replace global scaling and reduce repeated UI concerns and values.
8. Refresh documentation, assets, and developer tooling.

## Planned work, not technical debt

MinIO uploads, ingestion-job claiming, evidence normalisation, provenance, and
derived observations were next-product milestones rather than repairs to
shipped behaviour. The P1 foundations needed resolving before that work began.
